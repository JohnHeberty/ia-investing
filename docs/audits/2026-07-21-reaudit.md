# Reauditoria do IA Investing

Data da análise: 21 de julho de 2026  
Branch observado: `main`  
Commit mais recente observado: `69f7e13`

## Limites da verificação

A análise foi feita sobre o repositório público e arquivos acessíveis individualmente.
O ambiente desta sessão não conseguiu clonar o repositório completo e não havia uma
integração GitHub com permissão de escrita. Portanto:

- não houve push, commit ou alteração do `main`;
- não foi possível executar de forma independente a suíte completa;
- afirmações de testes passando foram tratadas como declarações do commit, não como
  evidência reproduzida nesta auditoria;
- a conclusão é uma auditoria estática de composição, contratos e prontidão.

## Conclusão executiva

O projeto evoluiu muito em relação ao esqueleto inicial. Agora existem frontend,
migrations, contratos, domínio financeiro mais amplo, testes de integração, paper
execution, risco, evidências, teses, política/macro e vários componentes de painel.
Isso é progresso real.

Entretanto, o estado atual ainda não é coerente como runtime de produção. Há duas
arquiteturas simultâneas:

1. uma arquitetura canônica nova sob `src/ia_investing`;
2. entrypoints e pacotes legados sob `src/apps`, `src/agents`, `src/database`,
   `src/workflows` e outros roots.

O resultado é que uma parte relevante do que foi implementado pode existir no código,
mas não está necessariamente composta no processo que realmente sobe API e workers.
O sistema está mais próximo de um **protótipo institucional amplo** do que de uma
plataforma financeira liberável.

# Achados prioritários

## P0-01 — Colisão do namespace `agents`

### Evidência

Existem simultaneamente:

```text
src/agents
src/ia_investing/ai
```

O código legado e o provider canônico importam:

```python
from agents import Agent, Runner
```

O `pyproject.toml` descobre todos os pacotes sob `src` com `include = ["*"]`.

### Risco

Ao resolver `agents`, Python pode encontrar o pacote local antes do SDK instalado.
Isso pode causar import circular, atributos ausentes ou carregar o código errado. O
provider mock pode esconder o problema; ele aparece justamente ao ativar OpenAI real.

### Correção

1. Reescrever todos os imports locais para `ia_investing.ai`.
2. Migrar quaisquer classes ainda exclusivas do legado.
3. Excluir `src/agents`.
4. Restringir package discovery.
5. Adicionar teste de resolução de módulo.
6. Executar smoke test com provider real em ambiente não produtivo.

### Aceite

```python
import agents
assert "site-packages" in agents.__file__
```

ou equivalente independente do sistema operacional; nenhum caminho deve apontar para
o repositório.

## P0-02 — API sobe a arquitetura antiga

O `src/apps/api/main.py` observado possui 37 linhas, importa diretamente os routers
antigos e inclui somente emissores, financeiros, portfolio, agents e health. Não há
composição visível de OIDC, autorização, telemetry, route registry v1 ou novos domínios.

### Consequência

Arquivos e endpoints novos podem existir, mas o processo oficial não os expõe. O
frontend pode então operar com mocks, dados parciais, rotas não montadas ou contratos
incompatíveis.

### Correção

Criar `create_app(settings)` e um registro explícito de routers. O teste deve comparar
os routers esperados com os realmente montados. Autenticação deve ser global e cada
command precisa de policy própria.

## P0-03 — Worker Temporal sem activities

O worker observado registra quatro workflows e nenhuma activity. Activities são o
local apropriado para I/O, banco, HTTP, agents e cálculos não determinísticos. Sem
registro, execuções falham quando tentam agendar uma activity.

### Correção

Criar registries por capacidade, registrados por `settings.worker.capability`, com
workflows e activities exatos. Adicionar retries, timeouts, heartbeat e tradução de
erros retryable/non-retryable.

## P0-04 — Scheduler em memória não executa workflows duráveis

O scheduler ainda usa `_last_run`, `while True` e `sleep(60)`. A ingestão CVM apenas
instancia a classe do workflow e escreve que ela foi carregada; notícias apenas geram
log. O estado é perdido em restart e não há overlap, catch-up ou backfill.

### Correção

Substituir por Temporal Schedules provisionados idempotentemente. Remover o processo
scheduler depois da migração.

## P0-05 — Segurança configurada, mas não composta

Há settings OIDC, porém o entrypoint da API observado não aplica autenticação. Ter
campos de configuração não equivale a proteção.

### Correção

- OIDC/JWT validation;
- contexto de organização e ator;
- RBAC+ABAC;
- deny-by-default;
- políticas por portfolio e estado;
- audit event para mutações;
- testes negativos de autorização;
- segregação de funções para comitê, risco e operações.

## P0-06 — CI/CD removido

O histórico registra a remoção deliberada do workflow de CI. Em software financeiro,
CI é parte do controle operacional, não uma conveniência.

### Consequência

Não há garantia contínua de que:

- migrations aplicam;
- testes continuam passando;
- imports não colidem;
- OpenAPI não quebra;
- frontend compila;
- workflows continuam replay-safe;
- prompts mantêm score mínimo;
- dependências não introduzem vulnerabilidades.

### Correção

Restaurar checks obrigatórios e proteção de branch. O workflow incluído neste kit é
um ponto de partida, a ser ajustado aos scripts reais do frontend.

## P0-07 — Dados políticos sintéticos usados como entrega funcional

O commit mais recente declara 12 outcomes políticos sintéticos. Isso é adequado para
testes e demonstração, mas não para calibrar probabilidade de tramitação, aprovação ou
impacto econômico em produção.

### Correção

- marcar origem e elegibilidade;
- separar fixtures de produção;
- criar dataset histórico oficial, versionado e auditado;
- calibrar por tipo, casa, regime de tramitação, tema e período;
- medir Brier score, log loss e calibration curves;
- bloquear readiness se o modelo depender só de sintético.

## P0-08 — Implementado não significa integrado

O commit mais recente adiciona dezenas de arquivos, endpoints e testes, porém o API
entrypoint e o worker continuam antigos. Isso é um problema de **composition root**.

### Correção

Criar testes E2E iniciando os mesmos comandos usados por Docker/produção e validando:

```text
HTTP -> application service -> workflow -> activity -> DB/object store -> query -> web
```

Não basta importar serviços isoladamente.

# Achados P1

## P1-01 — Namespace duplo além de `agents`

Há sinais de pacotes legacy e canônicos para banco, workflows, connectors, domain,
portfolio e backtesting. Isso cria drift e aumenta o risco de corrigir uma versão que
não é a usada em produção.

**Decisão:** `ia_investing` deve ser a única implementação. Compatibilidade deve ter
prazo e telemetria de uso.

## P1-02 — Falhas de provider podem ser absorvidas como resultado

O runtime deve distinguir:

```text
business_rejection
validation_failure
retryable_infrastructure_failure
non_retryable_configuration_failure
```

Erros retryable precisam chegar ao Temporal como retryable. Retornar `status=failed`
como output normal impede retry e pode fazer o workflow continuar com análise ausente.

## P1-03 — Guardrails baseados principalmente em regex

Regex simples ajuda, mas é insuficiente para prompt injection em PDFs/HTML, conteúdo
ofuscado, outras línguas e instruções sem frases clássicas.

Camadas necessárias:

1. conteúdo externo encapsulado e rotulado como não confiável;
2. agent sem ferramentas destrutivas;
3. allowlist de fontes/ferramentas;
4. schema estrito;
5. verificação de evidência;
6. guardrail semântico;
7. aprovação em tool calls sensíveis;
8. testes adversariais multilíngues;
9. logs de trips;
10. isolamento entre análise e execução.

## P1-04 — README está obsoleto

A documentação ainda descreve `packages/` e comandos `mypy packages/ apps/`, embora o
código atual esteja em `src` e exista frontend em `web`.

O README precisa refletir somente comandos testados e a arquitetura ativa.

## P1-05 — Frontend amplo sem prova de integração real

O frontend contém páginas para assets, portfolios, agents, audit, backtests, committee,
data quality, macro, opportunities, paper, policy e risk, além de Storybook e testes.
Isso é um bom avanço.

A pendência é provar que cada tela:

- consome endpoint montado;
- respeita autenticação;
- trata loading/empty/stale/error;
- exibe `as_of`, fonte e qualidade;
- não usa mock em produção;
- atualiza por operação assíncrona;
- possui teste Playwright contra a stack real.

## P1-06 — Declarações de testes não são controle permanente

O commit afirma dezenas de testes passando. Sem CI, isso não impede regressão no commit
seguinte. O importante é o gate obrigatório, não a contagem textual no commit.

## P1-07 — Settings canônicos não garantem adoção

A estrutura de settings é boa: provider mock por default e validação de produção. Mas
entrypoints antigos importam outras configurações. É necessário eliminar fontes de
configuração concorrentes.

## P1-08 — Observabilidade precisa ser inicializada no runtime real

Exigir traces correlacionados para:

```text
request_id
workflow_id
activity_id
agent_run_id
research_case_id
thesis_version_id
portfolio_version_id
rebalance_proposal_id
order_id
```

Logs ou módulos de telemetry sem inicialização no composition root não atendem esse
requisito.

# Critério de "condizente"

O sistema poderá ser considerado condizente com a proposta quando demonstrar, em uma
execução reproduzível, este caminho completo:

1. uma fonte oficial publica informação;
2. o schedule durável inicia o workflow;
3. connector baixa e preserva versão/raw/hash;
4. parser extrai sem transformar ausência em zero;
5. qualidade promove ou quarentena;
6. métrica determinística é calculada com lineage;
7. agent analisa conteúdo não confiável sem ganhar autoridade de execução;
8. claims são validados contra evidências;
9. tese recebe nova versão e diff;
10. risco e valuation são recalculados;
11. portfolio architect pede uma otimização determinística;
12. committee pack é congelado;
13. risco/compliance/humano aprovam ou rejeitam;
14. paper execution gera ordens e fills simulados;
15. posições, caixa e NAV reconciliam;
16. dashboard mostra resultado, fontes, estado e timestamps;
17. auditor consegue reconstruir todo o caminho;
18. CI e observabilidade provam que o fluxo continua saudável.

No estado observado, vários elementos de 1 a 18 existem isoladamente, mas os P0 de
composition root, worker, scheduler, namespace, segurança e CI impedem afirmar que o
fluxo completo está operacional.
